# 1200 人并发方案

## 一、请求链路与瓶颈分析

```
1200 员工
  │  企微回调
  ▼
plugin_daemon（wecom-bot v3）     ← 瓶颈 1：单实例串行
  │  HTTP SSE
  ▼
Dify API（gunicorn + gevent）     ← 瓶颈 2：连接数限制
  │  HTTP
  ▼
DeepSeek V4 Flash                 ← ✅ 非瓶颈（2500 并发）
  │  部分请求
  ▼
RAGFlow（检索+embedding）         ← 瓶颈 3：Infinity 稳定性
```

### 各环节容量估算

| 环节 | 当前限制 | 吞吐量 | 1200 人够吗 |
|------|---------|--------|------------|
| plugin_daemon | 1 实例, 256MB, ~25 并发槽 | ~100 请求/分钟 | 正常够，极端排队 |
| Dify API | 1 worker gevent, 10 连接 | 10 连接/worker | 偏低 |
| DeepSeek | 2500 并发 | 远超需求 | ✅ 最不担心 |
| RAGFlow Infinity | 不稳定的单实例 | 峰值时假死 | ⚠️ 最大风险 |

### 并发计算

```
1200 人 × 1-2% 同时发消息 = 12-24 个并发请求
每个请求 LLM 耗时 ~15 秒
24 个请求 × (60s / 15s) = 96 请求/分钟
当前插件 ~25 并发槽 → 正常够用
```

极端情况（50% 人同时发消息 = 600 并发）→ 排队 1-2 分钟。概率极低。

## 二、分阶段方案

### 上线前（必须做）

#### 1. Dify API 连接数扩容

改 Dify 的 `api.env`：

```bash
# 改前
SERVER_WORKER_AMOUNT=1
SERVER_WORKER_CONNECTIONS=10

# 改后
SERVER_WORKER_AMOUNT=2
SERVER_WORKER_CONNECTIONS=50
```

- `WORKER_AMOUNT=2`：双 worker 冗余，一个挂另一个还能跑
- `CONNECTIONS=50`：每个 worker 同时处理 50 个请求，2 × 50 = 100 并发上限
- 改动方式：编辑 `~/dify/docker/envs/core-services/api.env`，重启 `dify-api-1`

#### 2. Infinity 5432 端口加固

改 RAGFlow 的 `docker-compose-base.yml`：

```yaml
# 改前
      - ${INFINITY_CS_PORT:-5432}:5432

# 改后
      - 127.0.0.1:${INFINITY_CS_PORT:-5432}:5432
```

然后重建 RAGFlow：
```bash
cd /opt/ragflow/docker && sudo docker compose down && sudo docker compose up -d
```

**原因**：Infinity 的 5432 暴露在外被安全扫描器/监控工具用非 PG 协议误连，导致 Infinity 假死 → RAGFlow 整个卡住。绑 127.0.0.1 后只有容器内 RAGFlow 能访问。

---

### 灰度期（观察 1-2 周后决定）

#### 3. 插件内存扩容

如果 `docker stats` 显示 wecom-bot 内存接近 256MB，改 manifest.yaml：

```yaml
# 改前
resource:
  memory: 268435456    # 256MB

# 改后
resource:
  memory: 536870912    # 512MB（翻倍）
  # 或
  memory: 1073741824   # 1GB
```

- 256MB → 512MB：并发槽 ~50
- 256MB → 1GB：并发槽 ~100
- 改动方式：编辑插件目录下的 `manifest.yaml`，重启 plugin_daemon

#### 4. 增加 wecom-bot 副本

如果流式轮询平均等待时间 > 10 秒，在 Dify 市场再装一两个 wecom-bot 副本，绑同一个 App。

- 2 个副本：并发槽 ~50（256MB 时）
- 4 个副本：并发槽 ~100

缺点：配置要维护多份，插件更新时每个副本都要部署。

---

### 长期稳定期（遇到问题才动）

#### 5. RAGFlow Infinity → Elasticsearch 迁移

如果 Infinity 持续不稳定（假死、索引丢失），切回 RAGFlow 默认的 ES：

- 改 `service_conf.yaml.template` 中注释掉 `infinity` 段，启用 `es` 段
- ES 已在 RAGFlow compose 中（`docker-es01-1`），无需额外部署
- 需要重建所有知识库索引（数据不丢，重新解析一遍即可）

#### 6. 独立文件存储

如果 MinIO 文件量大（如累计超过 500GB），考虑把 MinIO 迁出到独立服务器或云存储（S3/OSS）。

#### 7. RAGFlow 水平扩展

如果单 RAGFlow 检索吞吐不够，可部署多个 RAGFlow 实例（共享 MySQL/MinIO/ES），Dify 外部知识库后端配多个地址做负载均衡。

## 三、监控指标

上线后关注这些数值：

| 指标 | 命令 | 健康范围 |
|------|------|---------|
| plugin 内存 | `docker stats dify-plugin_daemon-1` | < 200MB |
| Dify API 内存 | `docker stats dify-api-1` | < 2GB |
| RAGFlow 内存 | `docker stats ragflow-ragflow-cpu-1` | < 10GB |
| 流式轮询等待 | 企微体验 | < 5 秒 |
| Infinity 稳定 | `docker logs ragflow-infinity-1 \| grep error` | 无新 error |
| 整体响应 | 用户反馈 | < 30 秒出回复 |

---

## 四、总结

| 优先级 | 改动 | 效果 | 风险 |
|--------|------|------|------|
| **P0** | Dify API 连接数 + Infinity 端口 | 瓶颈前移，防止假死 | 无 |
| **P1** | 插件内存扩容 | 更多并发槽 | 改完需重启 |
| **P2** | 多副本 | 水平扩展 | 维护成本 |
| **P3** | 切 ES | 搜索引擎稳定 | 需重建索引 |

P0 做完即可上线。P1-P3 灰度期按实际数据决定。
