### 3. RAGFlow Infinity 稳定性问题
- **根因分析**：Infinity 的 5432 端口暴露在服务器外网（0.0.0.0:5432），被安全扫描器用多种协议轰炸，导致 Infinity 假死

- **临时修复**：重启 Infinity 恢复
- **持久化方案待定**：需将 5432 端口改为绑定 127.0.0.1

## 7 月 21 日（周二）

### 1. nginx 配置持久化
- **问题**：`doc-download` 路由只能加在容器内的 `ragflow.conf`，但 RAGFlow 启动时 `entrypoint.sh` 会从模板 `ragflow.conf.python` 执行 `cp -f` 覆盖，容器重建后丢失
- **尝试路径**：直接改容器内文件（重建丢）→ 挂载 `ragflow.conf`（被 `cp -f` 覆盖）→ 改宿主机 `ragflow.https.conf`（文件名不匹配，未被容器引用）→ 改容器内模板文件（镜像自带，重建恢复原样）
- **最终方案**：利用 `entrypoint.sh` 的机制反向实现持久化——
  1. 把容器内模板 `ragflow.conf.python` 导出到宿主机 `./nginx/` 目录
  2. 在宿主机模板文件中加入 `doc-download` 路由
  3. docker-compose.yml 新增挂载 `./nginx/ragflow.conf.python:/etc/nginx/conf.d/ragflow.conf.python`
  4. 容器启动时 `entrypoint.sh` 执行 `cp -f` 的**源文件**就是宿主机已含 doc-download 的模板，生成的 `ragflow.conf` 自然包含下载路由
- **踩坑**：模板文件括号不匹配（多余/缺失 `}`），nginx 启动报 `"location" directive is not allowed here`；Redis 端口冲突（旧容器名 `ragflow-redis-1` 仍占用 6379）
- **验证**：修复后 `docker compose down && up -d`，nginx 启动正常，doc-download 配置自动生效，curl 测试 HTTP 200，访问日志确认多人下载成功

### 2. 1200 人并发容量评估
- **梳理请求链路**：企微 → plugin_daemon → Dify API → DeepSeek → RAGFlow
- **瓶颈定位**：
  - DeepSeek V4 Flash：2500 并发，非瓶颈
  - plugin_daemon：1 实例 256MB，估算可承载 ~25 并发 LLM 请求
  - Dify API：`SERVER_WORKER_CONNECTIONS=10`，当前配置偏低
- **扩容建议**：短期调大 `SERVER_WORKER_CONNECTIONS` 至 50、worker 至 2；中期将插件内存扩至 1GB；长期可多装 wecom-bot 副本
- **结论**：当前配置可满足 1200 人正常使用（1-2% 并发率），极端全员并发场景需排队但不会丢消息
